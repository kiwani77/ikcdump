1. Governance

35%

시험 최대 비중. CMDB 데이터의 장기적 신뢰성과 품질 관리를 검증합니다.

CMDB Health — 3C 모델

CMDB 건강도를 측정하는 세 가지 핵심 지표. 각각의 차이를 반드시 구분해야 합니다.

지표의미측정 대상예시
Completeness
완전성		필수 속성이 채워져 있는지Required / Recommended 필드Server에 OS 버전, IP 미입력
Correctness
정확성		데이터가 정확하고 최신인지Stale CI, Orphan CI, Duplicate CI60일 이상 미갱신 CI
Compliance
준수성		조직 정책/표준 준수 여부Audit 규칙, Desired StateProd 서버 16GB RAM 미충족
⚠ 시험 포인트
Staleness = Correctness 영역 (Completeness 아님!)
Orphan CI (부모 없는 Dependent CI) = Correctness 영역
Compliance는 Audit + Desired State 기반 논리적 조건 검사
CMDB Health Dashboard에서 3C 스코어 모니터링
Correctness 세부 유형
Stale CI—X일(기본 60일) 동안 업데이트되지 않은 CI. Audit 기반 감지
Orphan CI—부모(Parent)를 잃은 Dependent CI. 예: NIC의 Server 삭제됨
Duplicate CI—IRE 식별 규칙에 의해 동일하다고 판단된 2개 이상 CI
Logical Orphan—필수 데이터 논리적 누락. 예: VM인데 "Virtual" 플래그 미설정
Relationship Orphan—Dependent CI가 Parent와의 관계 상실
CMDB Data Manager

CMDB를 대규모로 관리하기 위한 정책(Policy) 기반 도구

Life Cycle 정책: CI의 Retire → Archive → Delete 흐름을 자동화
필터 조건: 특정 클래스/속성 값 기반으로 대상 CI 선별
일괄 처리: 대량 CI에 일관된 정책 적용
Attestation: CI 소유자에게 데이터 정확성을 주기적으로 확인 요청
⚠ 시험 포인트
"대규모 CMDB 관리" 시나리오 → Data Manager가 정답
Lifecycle: Operational → Non-Operational → Retired → Archived → Deleted
Data Manager vs 수동 수정 — 대량 작업은 항상 Data Manager 우선
중복 CI (Duplicate) 관리
De-duplication Task: IRE가 하나의 payload에서 2개 이상 매치 → 자동 생성
IRE는 레코드를 업데이트하지 않고 중단(Halt)
수동 검토 후 병합(Merge) 또는 유지 결정
De-duplication Wizard: 수동으로 중복 CI 식별 및 병합
Bulk Remediation: 대량 중복 일괄 처리
Remediation Playbook: 데이터 품질 개선 가이드라인/절차
CMDB Workspace & 역할
CMDB Workspace: Health Dashboard, Data Manager, Query Builder를 통합 제공
CMDB Manager: 전반적인 CMDB 거버넌스 책임자
Data Manager Admin: Data Manager 정책 관리
CI Owner: 특정 CI의 데이터 품질 책임
Data Certification & Ownership
데이터 인증: CI 데이터의 정확성을 공식적으로 검증하는 프로세스
소유권: "누가 이 CI를 관리하는가"를 명확히 정의
주기적 검토: CI Owner가 정기적으로 데이터를 검토하고 인증
핵심 원칙: 데이터 기반은 구조뿐 아니라 프로세스와 책임(Accountability)

2. Ingestion & IRE

19%

CMDB로 데이터가 들어오는 경로와 IRE의 동작을 이해해야 합니다.

IRE (Identification and Reconciliation Engine)

CMDB에 데이터가 들어올 때의 관문(Gatekeeper). 모든 데이터 소스에서 들어오는 데이터를 식별하고 조정합니다.

3. Insight

20%

거버넌스된 CMDB 데이터에서 가치와 가시성을 추출하는 영역입니다.

CMDB Query Builder
CI 간 복잡한 조인(Join)과 관계를 시각적으로 구축
여러 클래스를 넘나드는 쿼리 작성 가능
결과를 리포트/대시보드에 활용
관계(Relationship) 기반 필터링으로 영향 분석 활용
CMDB 360
단일 CI에 대한 360도 뷰 제공
관련된 모든 관계, 이벤트, 변경 이력을 한 화면에서 확인
건강도, 관계 맵, 연관 인시던트/변경 종합 파악
Dependency Map / Unified Map
CI 간 의존성(Dependency)을 시각적으로 표시
서비스 영향도 분석(Impact Analysis)에 핵심
Unified Map: CMDB 관계를 하나의 통합 뷰로 제공
장애 발생 시 영향받는 서비스/CI를 즉시 파악
NLQ (Natural Language Query)
자연어로 CMDB 데이터를 조회
예: "Show me all Windows servers in production"
기술적 쿼리 지식 없이도 데이터 접근 가능
CSDM Data Foundations Dashboard
CSDM 성숙도(Maturity Level) 측정 및 개선 도구
각 CSDM 도메인별 데이터 품질 점수 확인
부족한 도메인을 가시적으로 파악
지표(Indicators) & Analytics
Performance Analytics (PA): CMDB 건강도 추세를 시계열 분석
KPI: 목표 대비 CMDB 품질 측정
Dashboard: 실시간 CMDB 상태 모니터링
⚠ 시험 시나리오 매핑
"복잡한 관계 쿼리" → Query Builder
"단일 CI의 모든 정보를 한눈에" → CMDB 360
"영향도 분석" → Dependency Map
"CSDM 준수 수준 측정" → CSDM Data Foundations Dashboard

4. Configuration

15%

CMDB의 구조적 설정과 관리 방법을 다룹니다.

CI Class Manager
CMDB의 클래스 계층구조, 테이블, 속성 관리
클래스 추가/수정, 속성 설정, 계층 관리
Required 속성: System Dictionary에서 Mandatory 설정 (글로벌)
Recommended 속성: CI Class Manager에서 설정 (Soft Check)
⚠ Required vs Recommended 구분
Required: System Dictionary 설정, 글로벌 적용, 빈 값이면 저장 불가
Recommended: CI Class Manager 설정, 저장은 가능하지만 Completeness 감소
"CI가 저장 안 됨" → Required 미충족
"Health 스코어 떨어짐" → Recommended 미충족
CMDB 아키텍처 핵심
cmdb_ci: 모든 CI의 기본(Base) 테이블
cmdb_rel_ci: CI 간 관계(Relationship) 테이블
상속: 하위 클래스는 상위 클래스의 모든 속성을 상속
Principal Class: Reference 필드에서 보여주는 CI 범위 제한
예: Incident 폼의 CI 선택 시 Server, App, DB만 표시
관계 유형 (Relationship Types)
관계방향예시
Depends onA depends on BApplication depends on Server
Runs onA runs on BApp Server runs on VM
Hosted onA hosted on BDatabase hosted on Server
ContainsA contains BRack contains Server
Members ofA member of BServer member of Cluster
CI Reclassification
CI의 클래스를 변경하는 프로세스 (예: "Computer" → "Linux Server")
Discovery가 더 정확한 정보를 가져올 때 자동 발생 가능
IRE 규칙에 의해 관리됨

5. CSDM Fundamentals

11%

Common Service Data Model — 비즈니스 전략을 기술 구현에 연결하는 프레임워크. 비중 11%이지만 다른 영역과 밀접하게 연관됩니다.

CSDM 5.0 — 7개 도메인

모두 암기 필수! 특히 신규 추가된 Ideation & Strategy, Build & Integration

도메인설명핵심 테이블/개념
Ideation & Strategy비즈니스 전략/아이디어 수집Strategic Portfolio, Demand
Design서비스 설계/아키텍처 정의Business Capability, Information Object
Build & Integration서비스 구축 및 통합Application, Dev Environment
Deliver & Support서비스 전달/운영 지원Application Service, Business Service
Detect & Correct문제 감지 및 수정Incident, Problem, Change
Manage Portfolio포트폴리오 관리Business Application, Technical Service
FoundationCMDB 기본 인프라 데이터Hardware, Location, Network Gear
CSDM 성숙도 5단계
Foundation—CMDB 기본 설정, 기본 CI 클래스 정의, 데이터 소스 연결
Crawl—핵심 테이블 채우기. Business App, App Service 등 기본 매핑
Walk—관계 구축, 서비스 맵핑 확장, Health 모니터링 시작
Run—전체 CSDM 도메인 활용, 자동화된 거버넌스, KPI 기반 관리
Fly—완전한 CSDM 성숙, 전사적 데이터 기반 의사결정, 지속적 개선
Business Application vs Application Service
구분Business ApplicationApplication Service
테이블cmdb_ci_business_appcmdb_ci_service_auto
성격비운영
인벤토리/카탈로그 레코드		운영
실제 실행 중인 서비스
예시"Zoom" (도구 자체)"Zoom Production Instance"
목적포트폴리오 관리, 비용/소유권 추적모니터링, 인시던트 연결, 영향도 분석
도메인Manage PortfolioDeliver & Support
⚠ 시험 포인트
"비운영, 카탈로그 역할" → Business Application
"운영 중, 모니터링 대상" → Application Service
Business App은 여러 Application Service를 가질 수 있음 (1:N)
"올바른 CSDM 도메인을 매핑하라" 유형 빈출
CSDM 5.0의 7개 도메인 + 신규 레이어(Ideation, Build) 암기 필수
